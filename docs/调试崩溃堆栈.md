# Android Recovery Flutter 崩溃堆栈调试指南

## 概述

Android Recovery 环境中没有 debuggerd / tombstone 服务，Runner 崩溃时只能靠自带的信号处理器输出原始 PC 地址。
本文档通过一个**真实的 SIGABRT 崩溃案例**，演示从原始地址到定位源码行号、分析根因的完整流程。

---

## 1. 崩溃输出格式

Runner 崩溃时会打印三部分信息：

```txt
=== CRASH: Signal 6 (SIGABRT) ===         ← 信号类型（6=SIGABRT, 11=SIGSEGV）
Memory maps (flutter/runner):             ← 内存映射表
  5b646f4000-5b647ac000 r--p 00000000 ... /tmp/flutter/flutter-runner
  5b647af000-5b649a7000 r-xp 000b7000 ... /tmp/flutter/flutter-runner
  ...
  7392c5b000-739518c000 r--p 00000000 ... /tmp/flutter/libflutter_engine.so
  739519b000-7396ade000 r-xp 02530000 ... /tmp/flutter/libflutter_engine.so
  ...
Backtrace:                                ← PC 地址列表
  #0 pc 5b647de728 ???
  #1 pc 848 [vdso] (__kernel_rt_sigreturn)
  #2 pc 51514 /system/lib64/libc.so (abort)
  #3 pc 73951a6b74 ???
  ...
=== END BACKTRACE ===
```

**三个关键信息**：

| 字段 | 含义 | 用途 |
| ------ | ------ | ------ |
| Signal | 崩溃类型 | SIGABRT=主动 abort（如 `FML_CHECK` 失败），SIGSEGV=非法内存访问 |
| Memory maps | 各库在内存中的加载位置 | 计算 PC 地址到 ELF 文件偏移 |
| Backtrace | 崩溃时的调用栈 PC 地址 | 反解出函数名和源码行号 |

---

## 2. 第一步：提取基地址

从 Memory maps 中找到每个库的**第一个 `r--p` 映射（文件偏移为 `00000000`）**，其起始地址就是基地址。

以下面的真实崩溃为例：

```txt
Memory maps (flutter/runner):
  5b646f4000-5b647ac000 r--p 00000000 ... /tmp/flutter/flutter-runner     ← Runner 基地址
  5b647af000-5b649a7000 r-xp 000b7000 ... /tmp/flutter/flutter-runner
  ...
  7392c5b000-739518c000 r--p 00000000 ... /tmp/flutter/libflutter_engine.so  ← Engine 基地址
  739519b000-7396ade000 r-xp 02530000 ... /tmp/flutter/libflutter_engine.so
```

提取：

```txt
RUNNER_BASE=0x5b646f4000
ENGINE_BASE=0x7392c5b000
```

> ⚠️ **ASLR**：每次运行基地址都不同，必须从**同一次崩溃输出**中读取。

---

## 3. 第二步：计算文件偏移

`llvm-addr2line` 需要 ELF 文件内的偏移，不是运行时虚拟地址：

```txt
文件偏移 = PC地址 - 基地址
```

### 判断 PC 属于哪个库

看 Memory maps 中哪个范围包含该地址：

- `5b646f4000 ~ 5b649bf000` → `flutter-runner`
- `7392c5b000 ~ 7396bc9000` → `libflutter_engine.so`
- 不在任何已知范围内 → 可能是 JIT 代码、vdso 或未列出的库

### 批量计算示例

```bash
# 从 backtrace 中挑选属于 engine 的地址，减去 ENGINE_BASE
python3 -c "
base = 0x7392c5b000
for pc in [0x73951a6b74, 0x739527858c, 0x739564dde0, 0x739560cae4]:
    print(hex(pc - base))
"
# 输出：
# 0x254bb74
# 0x261d58c
# 0x29f2de0
# 0x29b1ae4
```

## 4. 第三步：用 llvm-addr2line 反解

### 基本命令

```bash
NDK="$HOME/Library/Android/sdk/ndk/28.2.13676358/toolchains/llvm/prebuilt/darwin-x86_64/bin"

"$NDK/llvm-addr2line" -C -f -e <ELF文件> <偏移1> <偏移2> ...
```

参数：

- `-C`：反修饰 C++ 符号名（demangle）
- `-f`：同时输出函数名
- `-e`：指定 ELF 文件（必须是**未 strip 的**带调试符号版本）

### 文件路径

```bash
# 引擎（409MB，带完整调试符号）
ENGINE_SO="flutter-3.29-gpu/engine/src/out/android_debug_unopt_arm64_embedder_vulkan/libflutter_engine.so"

# Runner（strip 前的副本，12MB，由 for.sh 自动保存）
RUNNER="flutter-embedded-linux/build/flutter-drm-dumb-backend.sym"
```

> Runner 部署到设备的是 2.7MB strip 版，`llvm-addr2line` 无法解析。
> `for.sh build-runner` 会在 strip 前自动保存 `.sym` 副本。

### 完整示例

```bash
NDK="$HOME/Library/Android/sdk/ndk/28.2.13676358/toolchains/llvm/prebuilt/darwin-x86_64/bin"
ENGINE_SO="flutter-3.29-gpu/engine/src/out/android_debug_unopt_arm64_embedder_vulkan/libflutter_engine.so"

# 反解 engine 地址
"$NDK/llvm-addr2line" -C -f -e "$ENGINE_SO" \
  0x254bb74 0x261d58c 0x29f2de0 0x29b1ae4 0x29b0338 0x29afec4 \
  0x29edd14 0x29edc00 0x34e6228 0x34e5840
```

```bash
./for.sh run   # Software 模式
```

### 5.6 定位崩溃源码

关键帧是 `#5` 和 `#6`：

```txt
#6  DrawTextFrameOp::dispatch → 将 drawTextFrame 操作分发给当前 Dispatcher
#5  DlSkCanvasDispatcher::drawTextFrame → dl_sk_dispatcher.cc:315
```

查看 `dl_sk_dispatcher.cc:315` 的源码：

```cpp
void DlSkCanvasDispatcher::drawTextFrame(
    const std::shared_ptr<impeller::TextFrame>& text_frame,
    DlScalar x, DlScalar y) {
  FML_CHECK(false);   // ← 无条件 abort！Skia 不支持 drawTextFrame
}
```

`FML_CHECK(false)` 是一个永远触发的断言 → `LOG(FATAL)` → `abort()` → SIGABRT。

### 5.7 根因分析

**调用链**（从 Dart 到崩溃点）：

```txt
Dart Paragraph.paint()
  → ParagraphBuilderSkia(impeller_enabled=true)   ← 问题根源
  → DisplayListParagraphPainter::drawTextBlob()
  → if (impeller_enabled_) → builder_->DrawTextFrame(...)   ← 录入了 Impeller 专用操作
  → ...
  → 回放 DisplayList 时: DlSkCanvasDispatcher::drawTextFrame()
  → FML_CHECK(false) → abort
```

**决策逻辑**在 `paragraph_skia.cc` 中：

```cpp
void DisplayListParagraphPainter::drawTextBlob(...) {
#ifdef IMPELLER_SUPPORTS_RENDERING
  if (impeller_enabled_) {
    builder_->DrawTextFrame(MakeTextFrameFromTextBlobSkia(blob), ...);  // Impeller 路径
    return;
  }
#endif
  builder_->DrawTextBlob(blob, ...);  // Skia 路径
}
```

两个条件控制走哪条路径：

| 条件 | 哪里决定 | 当前值 | 原因 |
| ------ | --------- | -------- | ------ |
| `IMPELLER_SUPPORTS_RENDERING` | 编译时 | **true** | 引擎编译了 `impeller_enable_vulkan=true` |
| `impeller_enabled_` | 运行时 | **true** | `settings.h` 对 Android 目标默认 `enable_impeller=true` |

**核心问题**：`settings.h` 中 Android 目标的默认值：

```cpp
// flutter/common/settings.h
#if FML_OS_ANDROID || FML_OS_IOS
  bool enable_impeller = true;    // ← Android 默认开启 Impeller
#else
  bool enable_impeller = false;
#endif
```

Runner 虽然知道自己在 Software 模式（`enable_impeller_=false`），但没有把这个信息传给引擎，导致引擎内部仍然按 Impeller 路径生成 display list 操作。

### 5.8 修复

在 `flutter_elinux_engine.cc` 的 `RunWithEntrypoint()` 中，当 Runner 为 Software 模式时，自动注入 `--enable-impeller=false`：

```cpp
std::vector<std::string> switches = project_->GetSwitches();

// Android 目标的引擎默认 enable_impeller=true（settings.h），
// 必须显式传 --enable-impeller=false 才能让引擎内部走 Skia DrawTextBlob 路径
if (!enable_impeller_) {
  bool has_impeller_switch = false;
  for (const auto& s : switches) {
    if (s.find("--enable-impeller") != std::string::npos) {
      has_impeller_switch = true;
      break;
    }
  }
  if (!has_impeller_switch) {
    switches.push_back("--enable-impeller=false");
  }
}
```

修复后 Software 模式正常渲染文本，不再崩溃。

## 6. 一键反解脚本

将以下内容保存为 `symbolize_crash.sh`，或直接在终端使用：

```bash
#!/bin/bash
# 用法: ./symbolize_crash.sh
# 交互式输入基地址和 PC 地址，自动计算偏移并反解

NDK="$HOME/Library/Android/sdk/ndk/28.2.13676358/toolchains/llvm/prebuilt/darwin-x86_64/bin"
PROJECT="$(cd "$(dirname "$0")" && pwd)"

# 引擎 .so（带完整调试符号）
ENGINE_SO="$PROJECT/flutter-3.29-gpu/engine/src/out/android_debug_unopt_arm64_embedder_vulkan/libflutter_engine.so"
# Runner（strip 前副本）
RUNNER="$PROJECT/flutter-embedded-linux/build/flutter-drm-dumb-backend.sym"

echo "=== Flutter Recovery 崩溃反解工具 ==="
echo ""
echo "从崩溃输出的 Memory maps 中提取基地址："
echo "  找 r--p 00000000 那行的起始地址"
echo ""
read -p "Engine 基地址 (hex, 如 0x7392c5b000): " ENGINE_BASE
read -p "Runner 基地址 (hex, 如 0x5b646f4000): " RUNNER_BASE
echo ""
echo "粘贴 Backtrace 的 PC 地址（每行一个，空行结束）："

declare -a engine_addrs
declare -a runner_addrs
declare -a unknown_frames

while IFS= read -r line; do
  [[ -z "$line" ]] && break
  # 提取 PC 地址
  pc=$(echo "$line" | grep -oE '0x[0-9a-fA-F]+|[0-9a-fA-F]{10,}' | head -1)
  [[ -z "$pc" ]] && continue

  # 跳过 vdso 和已知库
  echo "$line" | grep -q "vdso\|libc\.so" && continue

  pc_val=$((16#${pc#0x}))
  eb=$((16#${ENGINE_BASE#0x}))
  rb=$((16#${RUNNER_BASE#0x}))

  offset_e=$((pc_val - eb))
  offset_r=$((pc_val - rb))

  # 引擎: 0 ~ 64MB 范围
  if [[ $offset_e -ge 0 && $offset_e -lt 0x4000000 ]]; then
    engine_addrs+=("$(printf '0x%X' $offset_e)")
  # Runner: 0 ~ 16MB 范围
  elif [[ $offset_r -ge 0 && $offset_r -lt 0x1000000 ]]; then
    runner_addrs+=("$(printf '0x%X' $offset_r)")
  else
    unknown_frames+=("$pc")
  fi
done

if [[ ${#engine_addrs[@]} -gt 0 ]]; then
  echo ""
  echo "=== libflutter_engine.so (${#engine_addrs[@]} 帧) ==="
  "$NDK/llvm-addr2line" -C -f -e "$ENGINE_SO" "${engine_addrs[@]}"
fi

if [[ ${#runner_addrs[@]} -gt 0 ]]; then
  echo ""
  echo "=== flutter-runner (${#runner_addrs[@]} 帧) ==="
  if [[ -f "$RUNNER" ]]; then
    "$NDK/llvm-addr2line" -C -f -e "$RUNNER" "${runner_addrs[@]}"
  else
    echo "⚠️  未找到带符号的 Runner: $RUNNER"
    echo "   请先运行 ./for.sh build-runner 生成 .sym 副本"
    echo "   尝试用 strip 版本（可能输出 ??）："
    "$NDK/llvm-addr2line" -C -f -e "$PROJECT/flutter-embedded-linux/build/flutter-drm-dumb-backend" "${runner_addrs[@]}"
  fi
fi

if [[ ${#unknown_frames[@]} -gt 0 ]]; then
  echo ""
  echo "=== 未知帧（可能是 JIT 代码或其他库）==="
  printf "  %s\n" "${unknown_frames[@]}"
fi
```

## 7. 注意事项

### SIGABRT vs SIGSEGV

| 信号 | 含义 | 常见原因 |
|------|------|---------|
| SIGABRT (6) | 主动 abort | `FML_CHECK()` 失败、`LOG(FATAL)`、`assert()` |
| SIGSEGV (11) | 非法内存访问 | 空指针、野指针、栈溢出 |

SIGABRT 的关键特征：backtrace 中会出现 `libc.so (abort)` + `fml::LogMessage::~LogMessage()`。
搜索引擎源码中对应位置的 `FML_CHECK` / `FML_LOG(FATAL)` 即可找到断言条件。

### debug_unopt 构建的符号偏移不精确

`llvm-addr2line` 对未优化构建可能报告的行号有 1~3 行偏差。

**应对方法**：

- 看整条调用链（从底向上），确认函数名
- 用函数名回到源码搜索 `FML_CHECK` / `FML_LOG(FATAL)` 位置
- 对 SIGABRT，真正的断言位置通常在 `abort` 上方第 2~3 帧

### Runner 符号

| 文件 | 大小 | 符号 | 用途 |
|------|------|------|------|
| `build/flutter-drm-dumb-backend` | ~2.7MB | ❌ 已 strip | 部署到设备 |
| `build/flutter-drm-dumb-backend.sym` | ~12MB | ✅ 完整 | 崩溃反解 |

`.sym` 文件由 `./for.sh build-runner` 在 strip 前自动保存。
如果缺少 `.sym`，重新执行 `./for.sh build-runner` 即可生成。

### 引擎符号

| 构建类型 | 大小 | 可反解 |
| --------- | ------ | -------- |
| `debug_unopt`（当前使用） | ~409MB | ✅ 直接反解 |
| `release`（`stripped_symbols=false`） | ~60MB | ✅ 可反解 |
| `release`（`stripped_symbols=true`） | ~20MB | ❌ 无符号 |

当前项目的 debug_unopt 构建设置了 `stripped_symbols=false`，引擎 `.so` 可以直接反解。

### 地址不在已知库范围内

backtrace 末尾可能出现不属于任何已知库的 PC 地址（如 `0x73919eb5b8`）。
这通常是：

- Dart VM 的 JIT 编译代码
- 未列出的共享库（如 `libandroidicu.so`、`libsqlite.so`）
- vdso 虚拟内存

这些帧通常不是崩溃的关键路径，可以忽略。

## 9. 调试流程总结

```txt
崩溃发生
  │
  ├─ 1. 从输出中复制 Memory maps 和 Backtrace
  │
  ├─ 2. 提取基地址
  │      ENGINE_BASE = r--p 00000000 行的起始地址 (libflutter_engine.so)
  │      RUNNER_BASE = r--p 00000000 行的起始地址 (flutter-runner)
  │
  ├─ 3. 计算偏移
  │      偏移 = PC - 基地址
  │      先判断 PC 落在哪个库的地址范围内
  │
  ├─ 4. llvm-addr2line 反解
  │      用未 strip 的 .so/.sym 文件
  │      输出函数名 + 源码行号
  │
  ├─ 5. 从底向上读调用链
  │      找到关键帧（通常是 abort 上方 2~3 帧）
  │
  └─ 6. 查看源码定位根因
         SIGABRT → 搜索 FML_CHECK / FML_LOG(FATAL)
         SIGSEGV → 检查空指针 / 野指针 / 生命周期问题
```
