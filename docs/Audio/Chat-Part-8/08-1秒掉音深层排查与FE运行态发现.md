**为什么当时会被写成这样**
这通常是出于几个现实原因：

1. 先把视频播放跑通
   - 视频解码线程已经在跑，音频如果也跟它抢同一个 demux/decode 状态，复杂度会上升。

2. 先规避音视频时钟同步
   - 先落盘 WAV 后，音频线程只负责“顺序喂 PCM”，不需要先解决完整的音频时钟、缓冲欠载、重采样时序。

3. seek 实现便宜
   - 直接裁 WAV，比正确维护音频解码器、重采样器、环形缓冲、PTS 对齐要简单得多。

4. 先验证 AGM 输出链路
   - 先证明 agm_audio_player_write 可用，再谈实时流，是一种原型期写法。

所以这不是底层能力不足，而是当前实现目标偏“先能响、先能播”，不是你要的“实时流式音频架构”。

**如果按你的需求改，正确方向是什么**
要改成你要的模型，核心就是删掉“临时 WAV”这层，变成：

1. 单独保留音频解码状态
   - 音频流自己有 AVCodecContext、SwrContext、packet/frame 生命周期。

2. FFmpeg 实时解码
   - 从 demux 取音频包
   - 解码成 frame
   - 立即重采样成 AGM 需要的 PCM 格式

3. 直接写 AGM
   - 把重采样后的 PCM 直接送进 agm_audio_player_write
   - 中间最多只保留一个内存环形缓冲，不再写磁盘文件

4. seek 时真正刷新音频流水线
   - av_seek_frame
   - avcodec_flush_buffers
   - swr 状态清空
   - PCM 缓冲清空
   - 从新时间戳继续解码并写入 AGM
