核心可以分成这几块：

**1. 先把 recovery 音频底座补起来**
这一步最花时间，而且不是 C 层问题。

做的事主要是：

1. 把 vendor audio userspace 链拉起来
   - `vndservicemanager`
   - `pd-mapper`
   - `per_mgr / pm-service`
   - `adsprpcd`
   - `audioadsprpcd`
   - `audiohalservice.qti`

2. 处理 recovery 下 HAL 起不来的问题
   - 一开始不是播放器不行，而是 HAL 在 recovery 里根本没走到正常 full parse 路径
   - 需要改 init / rc / wrapper，让它不要停在 stub path

3. 让 recovery 真能看到正确的音频运行环境
   - 挂对 `vendor / odm / system`
   - 补对 `LD_LIBRARY_PATH`
   - 让 recovery 能读到 `backend_conf.xml`
   - 让 recovery 能读到 `ftm_test_config`
   - 让 recovery 能读到 usecase / kv / mixer 相关 xml

这一步做完之前，谈 C 播放没有意义，因为 graph 根本没法稳定起来。

**2. 把 kernel / machine / 声卡这一层打通**
这一步也非常耗时。

做的事包括：

1. 追 `sound card` 为什么没出来
2. 补 machine / codec / macro / supplier / deferred probe 链
3. 处理 `aw882xx`、`wcd`、`macro`、`spf_core_platform` 这些前置
4. 确认 `/dev/snd` 和对应 `pcmC0D*` 节点真的出现
5. 确认不是“用户态命令成功”，而是内核里这条链真的注册完成

这一步不通，用户态所有播放器都只是在空转。

**3. 把“厂商成功路径”拆开看懂**
真正能出声的不是普通 `tinyalsa + pcm_write`，而是厂商那条已经成功的路径。

所以花了很多时间在：

1. 实测 `tinyplay_ftm`
2. 实测 `agmplay`
3. 提取 stock 的 `tinymix / tinyplay / tinypcminfo / tinycap`
4. 观察它们和普通 tinyalsa 路线的差异
5. 确认：
   - 普通 `tinyplay_stock` 只能把流推到一部分路径
