User: 你从0到出声音，给我以一个大的视角，告诉我如何一步步完成的

GitHub Copilot: 如果从大的视角看，这件事不是“写了一个播放器然后它响了”，而是把一条被截断了很多层的音频链，逐层补齐，最后才让纯 C 数据真正进到扬声器。

可以把整个过程分成 7 个阶段来看。

**一、先判断问题到底在什么层**
一开始最容易误判的是，把“没声音”理解成单纯的 mixer 没开，或者播放器参数不对。

但实际往下拆后，很快发现不是这么简单：

1. recovery 里不只是末端扬声器状态有问题
2. 更早的音频服务链、AGM/PAL 注册链、graph 建立链本身也不完整
3. 所以当时就算手工开一些 WSA、Awinic 控件，也不能直接推出一定会响

这一步的意义是先建立分层模型：

1. 内核和声卡层有没有起来
2. audio HAL 和 AGM/PAL 服务有没有真正注册
3. AGM graph 能不能建立
4. graph 建立后，speaker/PA 末端状态有没有放开
5. 用户态播放器是不是按正确顺序启动流

只有把问题拆成这几层，后面每次推进才不会乱。

**二、先把 recovery 的底座补到“能进入 AGM 世界”**
这一步解决的是“连跑到 AGM graph 层的资格都没有”。

做的核心事情有两类。

第一类是内核和设备侧基础依赖：

1. 补齐声卡起来所需的模块链
2. 把之前导致声卡一直 defer 的缺项补上
3. 让 recovery 真正出现可用的音频相关设备和路径

第二类是用户态服务链：

1. 让 audiohalservice.qti 不再走错误或残缺路径
2. 让 AGM/PAL 相关服务真正注册成功
3. 把问题从“服务根本不存在”推进到“服务在，但 graph 还没完全建立”

这一步之后，标志性的变化不是“听到声音”，而是失败层级前移了：

1. 从找不到 IAGM 之类的服务错误
2. 变成能进到 graph_open、card100、MFC 这类 AGM 内部层级

这说明方向对了，因为系统终于开始在正确的层上失败。

**三、把可闻链分成两半：graph 层和 speaker 层**
这一步非常关键。因为后来反复验证后，已经能确定当前机器不是单层模型，而是两层同时成立才会响：

1. card100 这一侧负责 AGM graph、backend、MFC、stream 建立
2. card0 这一侧负责 WSA、WSA2、Awinic 功放、mute、volume 等末端 speaker 状态

这意味着：

1. 就算 graph_open 成功，也不代表一定可闻
2. 就算末端功放开了，如果 graph 没立起来，也不会响

所以后来真正有效的方法都必须同时覆盖这两层：

1. 上层让流进入正确的 AGM 路径
2. 下层把 speaker 和功放链真正放开

这一步其实把后面所有调试都变简单了，因为知道不能再只盯一个面。

**四、先用厂商成功路径当“参考答案”**
在 recovery 这种复杂环境里，最怕的是闭门猜协议。

所以中间做了一个非常正确的策略切换：不再先猜，而是先拿厂商自己的成功路径做参考答案。

这里最重要的是 tinyplay_ftm 这条链。

它提供了三类关键信息：

1. 哪个 backend 名是真的有效
2. 哪组 stream、device、devicepp、instance 元数据是真正被成功路径使用的
3. 成功播放时，用户态和 service 侧的调用时序到底是什么样

这一步带来的变化很大，因为它把很多错误方向排掉了：

1. 不是任意 backend 都能用
2. 不是随便猜一个 kv 都行
3. 不是“只要 graph_open 了就够了”
4. 成功链里确实存在 graph_start 这个明确阶段

也就是说，厂商链不是最终答案，但它给了你正确的骨架。

**五、把问题从 Flutter/FFI 容器里剥离出来，改成独立纯 C**
这一步解决的是“到底是框架问题，还是音频问题”。

你要求很明确：

1. 不要 FFI
2. 不要最终依赖厂商播放器二进制
3. 要纯 C demo
4. 要真正出声

所以后面做的核心动作，就是把播放路径收缩成独立纯 C 程序，把 Flutter、FFI、上层容器全部排除掉。

这一步的意义不只是实现偏好，而是技术上把问题隔离干净了：

1. 如果纯 C 和 Flutter 表现一致，那问题不在 FFI
2. 如果纯 C 也卡住，那就说明缺口就在 AGM、tinyalsa、metadata、stream start 这些底层
3. 这样每次修改都可以直接看最小系统的结果，不会被框架噪声干扰

后来事实也证明，这个判断完全正确。纯 C 和之前的阻塞边界一致，说明核心问题一直都在音频链本身。

**六、纯 C 主线真正推进的过程：从完全打不开，到 graph_open 成功**
这一阶段本质上是在复刻 tinyplay_ftm 的关键用户态准备链，但不是照抄二进制，而是把必要动作一点点还原出来。

这里经历了几个重要里程碑。

第一步，metadata 方向收敛：

1. 把一些错误猜法逐步排掉
2. 把更接近真实路径的几段 metadata 形状确认下来
3. 关键 payload 和顺序逐渐对齐到厂商链

第二步，tagged info 和 MFC 打通：

1. 纯 C 里能读到 getTaggedInfo
2. 能拿到目标 tag 对应的信息
3. 能拿到 MFC 的 miid
4. 能成功 configure_mfc

第三步，open 阶段突破：

1. graph_open ret 0 出现
2. pcm_open 成功
3. 说明 graph 已经不是完全没立起来了

到这里其实已经完成了 90% 的工程量。因为此时系统已经具备：

1. 正确 metadata
2. 正确 connect
3. 正确 tagged info
4. 正确 MFC
5. 正确 pcm_open

如果还没声，那只可能是最后启动时序还缺一脚。

**七、最后真正让它响的关键，不是新 metadata，而是显式启动流**
这是整个链路里最值得记住的结论。

当时纯 C 已经能做到：

1. graph_open ret 0
2. pcm_open 成功
3. MFC 成功
4. 但一写数据就固定在第 5 个 chunk 左右报错
5. 错误是 -131

表面上看很像“格式还不对”或者“metadata 还差一点”。

但真正往下对照后发现，关键差异是：

1. 成功的厂商链里，graph_open 之后还会出现 graph_start ret 0
2. 当前纯 C 链当时没有这个阶段
3. 也就是说流虽然 ready 了，但没有真正 start

于是最后做的改动非常小，但非常关键：

1. 在 pcm_open 成功后
2. 不立刻 pcm_write
3. 先显式调用 pcm_prepare
4. 再显式调用 pcm_start
5. 然后才进入写数据循环

这一步一补上，整个系统的状态就完全变了：

1. 用户态看到 pcm_prepare rc=0 和 pcm_start rc=0
2. service 侧第一次稳定出现 graph_start ret 0
3. aw882xx_start_pa 进入 start success
4. 2 秒、5 秒、8 秒测试音都能完整写完
5. 最终用户肉耳确认已经听到声音

这一步其实是整个项目从“快成功了”到“真的成功了”的分水岭。

**如果把整件事压缩成一条总路线**
从 0 到出声音，真正完成的是这条链：

1. 先补 recovery 的音频底座，让声卡、模块、服务注册链起来
2. 再把问题推进到 AGM graph 层，而不是停在更早的服务缺失层
3. 再把 speaker 末端 mixer 和功放状态补齐
4. 再用厂商成功路径提取真实 backend、metadata、时序
5. 再把播放路径改成独立纯 C，排掉 Flutter/FFI 噪声
6. 再把 pure C 主线推进到 graph_open、tagged info、MFC、pcm_open 成功
7. 最后补上 pcm_prepare 和 pcm_start，让 graph 从 open 进入 start
8. 数据真正流到功放，用户听到声音

**为什么这条路是对的**
因为每一步都不是盲改，而是把失败边界往后推：

1. 从服务不存在
2. 到 graph_open -19
3. 到 connect failed -5
4. 到 graph_open ret 0
5. 到 pcm_open 成功
6. 到 pcm_write -131
7. 到 graph_start ret 0
8. 到实机可闻

这是一条很典型的系统 bring-up 路线：不是一次性猜对，而是不断把失败层级后移，直到最后没有剩余缺口。

**现在回头看，最核心的三个认知**
如果只保留三个最高层的结论，我会保留这三个：

1. 这不是单纯播放器问题，而是 recovery 音频全链路 bring-up 问题
2. 这不是单层模型，必须同时打通 AGM graph 层和 speaker/PA 末端层
3. 纯 C 最后真正差的不是 metadata，而是 pcm_open 之后少了显式 prepare 和 start

**一句话总结**
从 0 到出声音，本质上是先把 recovery 的音频系统拉到“能建立 AGM graph”，再把 speaker 和功放末端放开，最后在纯 C 播放器里补上流启动时序，让 graph 从 open 真正走到 start，于是数据才第一次稳定变成了可闻声音。
