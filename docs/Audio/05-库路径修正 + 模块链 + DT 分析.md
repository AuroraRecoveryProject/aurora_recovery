# 05-库路径修正 + 模块链 + DT 分析 — 结构化总结

> 原始行范围：2595-3848
> 前情：compacted conversation 后继续。用户纠正了库文件替换的错误做法，用 `LD_LIBRARY_PATH` 优先指向真实 system 分区后，三段音频服务全部稳定驻留。修复 `/odm` 符号链接环。发现 clean recovery 下 `frpc_adsprpc` 默认不加载。补整条音频模块链：`oplus_audio_daemon` → `audpkt_ion` → `audio_prm` → `audio_pkt` → `frpc-adsprpc` → `machine_dlkm` + wcd/lpass/wsa。`/dev/aud_pasthru_adsp` 出现但声卡仍无。发现 `spf_core_platform` 的 DT 子节点 `sound` 存在但未实例化，正常系统里 `soc:spf_core_platform:sound` 子设备真实存在并已绑定 `sun-asoc-snd`。BoardConfig 补入最小音频模块集。用户说装备有 root 权限。

## 1. 检查 rpmsg/FastRPC 端点绑定状态

**原因**：继续沿 fastRPC 线推进，核对设备当前 rpmsg/fastrpc 绑定状态，确认下一步该直接补哪个运行时依赖。

先读取仓库记忆，然后确认设备连接状态，检查 rpmsg 总线和驱动绑定：

```bash
adb devices -l && adb -s 70a91f89 shell 'ls /dev | grep fastrpc; echo SEP; ls /sys/bus/rpmsg/devices; echo SEP; ls /sys/bus/rpmsg/drivers; echo SEP; for d in /sys/bus/rpmsg/devices/*; do echo X:$d; cat $d/name 2>/dev/null; readlink $d/driver 2>/dev/null; done'
```

ADSP 侧的 fastrpc rpmsg 端点已经出现，但没有绑定任何驱动。进一步查看该端点的具体属性——确认它是否带有设备树兼容串，以及内核是否有对应驱动：

```bash
adb -s 70a91f89 shell 'd=/sys/bus/rpmsg/devices/3000000.remoteproc-adsp:glink-edge.fastrpcglink-apps-dsp.-1.-1; ls $d; echo SEP; cat $d/name; echo SEP; cat $d/modalias 2>/dev/null; echo SEP; cat $d/uevent 2>/dev/null; echo SEP; ls /sys/bus/rpmsg/drivers/rpmsg_chrdev; echo SEP; cat /sys/bus/rpmsg/drivers/rpmsg_chrdev/uevent 2>/dev/null'
```

检查 dmesg 中是否有 fastrpc 相关错误：

```bash
adb -s 70a91f89 shell 'dmesg | grep -i fastrpc | tail -n 80'
```

**结论**：ADSP 侧的 rpmsg 端点带有设备树兼容串 `qcom,fastrpc`，但当前内核的 rpmsg 总线上没有对应的 fastrpc 驱动，只有通用的 `rpmsg_chrdev`。需要继续核对内核模块和设备节点创建路径，避免花时间在错误的绑定对象上。

## 2. 确认内核与用户态 fastrpc/adsprpc 驱动全貌

确认是内核模块没加载还是驱动根本没装进 recovery，检查已加载模块、dev 节点、sysfs 中 fastrpc 痕迹。

```bash
adb -s 70a91f89 shell 'cat /proc/modules | grep -Ei "fastrpc|adsprpc|rpc"'
```

```bash
adb -s 70a91f89 shell 'ls -l /dev/rpmsg* /dev/fastrpc* /dev/adsprpc* 2>/dev/null'
```

```bash
adb -s 70a91f89 shell 'find /sys -maxdepth 5 \( -iname "*fastrpc*" -o -iname "*adsprpc*" \) 2>/dev/null | head -n 120'
```

内核里看不到任何 fastrpc 或 adsprpc 驱动模块，只有一个 adsprpc 内存保留区平台设备和未绑定的 rpmsg 端点。继续查平台总线层——看驱动是不是根本没装进 recovery：

```bash
adb -s 70a91f89 shell 'ls /sys/bus/platform/drivers | grep -Ei "fastrpc|adsprpc|rpc"'
```

```bash
adb -s 70a91f89 shell 'd=/sys/bus/platform/devices/soc:qcom,msm-adsprpc-mem; ls $d; echo SEP; cat $d/modalias 2>/dev/null; echo SEP; readlink $d/driver 2>/dev/null; echo SEP; cat $d/uevent 2>/dev/null'
```

查用户态是否有 adsprpc 相关的库和可执行文件：

```bash
adb -s 70a91f89 shell 'ls /vendor/lib64 | grep -Ei "adsprpc|fastrpc|listener|audioadsp"; echo SEP; ls /vendor/bin /vendor/bin/hw 2>/dev/null | grep -Ei "audioadsp|adsprpc|audiohal|audioserver"'
```

**结论**：内核里没有任何 fastrpc/adsprpc 驱动模块已加载。`soc:qcom,msm-adsprpc-mem` 平台设备存在但无驱动绑定。当前更像是 recovery 内核没把 adsprpc/fastrpc 驱动带起来，而不是单纯缺用户态库。

## 3. 查找并加载 frpc-adsprpc.ko

**原因**：在设备上找 FastRPC 内核模块，确认依赖后手工加载。

搜索模块文件、确认分区状态：

```bash

```

```bash
adb -s 70a91f89 shell 'ls -l /dev/block/by-name | grep -E "vendor(_a|_b)?|odm(_a|_b)?|vendor_dlkm|odm_dlkm"'
```

```bash
adb -s 70a91f89 shell 'mount | grep -E "vendor|odm|dlkm"'
```

找到关键模块了：vendor_dlkm 里有 frpc-adsprpc.ko，但它现在没加载；我先查依赖，再直接尝试加载，这一步最有可能把缺失的 fastRPC 设备链拉起来。

```bash
adb -s 70a91f89 shell 'grep -n "frpc-adsprpc\.ko" /vendor_dlkm/lib/modules/modules.dep /vendor_dlkm/lib/modules/modules.softdep 2>/dev/null'
```

```bash
adb -s 70a91f89 shell 'modprobe -n -v frpc-adsprpc 2>/dev/null || busybox modprobe -n -v frpc-adsprpc 2>/dev/null || true'
```

```bash
adb -s 70a91f89 shell 'ls /vendor_dlkm/lib/modules/frpc-adsprpc.ko && cat /proc/modules | grep frpc-adsprpc'
```

依赖已经清楚了，模块本体也确实没加载；接下来直接用现成的 modules.dep 路径关系去加载它，看看是不是能把 rpmsg 端点接成设备节点。

```bash
adb -s 70a91f89 shell 'ls -ld /vendor/lib/modules /vendor_dlkm/lib/modules; readlink /vendor/lib/modules 2>/dev/null; echo SEP; modprobe -D -v -d / frpc-adsprpc 2>&1 | head -n 80'
```

```bash
adb -s 70a91f89 shell 'modprobe -v -d / frpc-adsprpc 2>&1; echo RET:$?; ls /dev | grep -E "fastrpc|adsprpc|rpmsg"; echo SEP; cat /proc/modules | grep frpc-adsprpc'
```

```bash
adb -s 70a91f89 shell 'dmesg | tail -n 120'
```

`modprobe` 在这个 recovery 里走不通，因为模块目录没有映射到 `/vendor/lib/modules`。日志里无额外线索，改用直接 `insmod`：

```bash
adb -s 70a91f89 shell 'insmod /vendor_dlkm/lib/modules/frpc-adsprpc.ko 2>&1; echo RET:$?; cat /proc/modules | grep frpc-adsprpc; echo SEP; ls /dev | grep -E "fastrpc|adsprpc|rpmsg"'
```

```bash
adb -s 70a91f89 shell 'dmesg | tail -n 80'
```

**结论**：`/vendor_dlkm/lib/modules/frpc-adsprpc.ko` 通过 `insmod` 成功加载，`/dev/fastrpc-adsp-secure` 出现——内核侧 fastRPC 设备注册路径已打通。下一步回到用户态链路，重跑 audio RPC 进程看 ASoC 是否开始枚举声卡。

## 45. 本阶段总结

### 正常系统基线三件关键事实

1. 这台机正常系统里**没有** `/dev/fastrpc-adsp` 或 `/dev/adsprpc-smd`——之前把它们当成硬前提不成立。正常系统实际存在的是 `/dev/fastrpc-adsp-secure`、`/dev/fastrpc-cdsp`、`/dev/fastrpc-cdsp-secure`，外加关键节点 `/dev/aud_pasthru_adsp`。
2. 正常系统声卡已枚举为 `sun-mtp-snd-card`，`/dev/snd` 下有完整的 `controlC0` 和 `pcmC0D*`。
3. 三进程的 SELinux 域确认为 `media:vendor_audioadsprpcd`、`system:vendor_adsprpcd`、`audioserver:hal_audio_default`。

### 回到 recovery 后确认的两件要事

1. `/dev/aud_pasthru_adsp` 不是单纯缺 dev 节点，而是连 `/sys/devices/virtual/aud_pasthru_adsp` 和主设备号 456 都不存在——整条注册路径没起来。
2. recovery SELinux 侧不完整：`vendor_audioadsprpcd`、`vendor_adsprpcd` 域均 `Invalid argument`，日志确认 `vendor_audioadsprpcd_exec` 未映射。

### 当前更准确的根因

不是"缺音频 DT"或"缺 `/dev/fastrpc-adsp`"，而是 recovery **同时缺两样关键前提**：
1. `aud_pasthru_adsp` 设备注册链路未启动
2. 正常系统音频 sepolicy 域映射缺失

### 下一步方向

1. 先临时 `setenforce 0`，再重跑 `adsprpcd`、`audioadsprpcd`、`audiohalservice.qti`，看 `aud_pasthru`/声卡是否推进
2. 找匹配的内核源码或已编译对象，追 `aud_pasthru_adsp` 注册驱动及 recovery 未触发原因
