## 三份文档总结

### 一、整体背景

这是一个在 TWRP Recovery 环境下手动 bring-up 音频输出的长期排查过程。目标设备使用高通平台（AGM/PAL 音频架构），配备 6 颗 Awinic AW882xx 智能功放（`4-0034/35/36`、`9-0034/35/36`），声卡为 `sun-mtp-snd-card`，后端接口为 `TDM-LPAIF-RX-SECONDARY`，backend 配置为 `48k/8ch/S32`。

### 二、核心问题演变

| 阶段 | 现象 | 根因 |
|---|---|---|
| 早期 | 完全无声 | 缺 soundtrigger VINTF fragment → audiohalservice.qti 反复重启 |
| 中期 | 约 1 秒可闻 + 白噪音 | FE PCM 按 `2ch/S16` 打开，backend 按 `8ch/S32` 建链 → 格式错配 |
| 中期修复后 | 白噪音消除，仍只有约 1 秒可闻 | 缺 `helper_connect + tagged_info + post-connect MFC` 三步 |
| 补齐 MFC 后 | 仍只有约 1 秒可闻 | AW 多芯片协同运行态模板整体从"可闻接力层"漂移到"低贡献肩部/过门态" |
| 2026-03-25 回退期 | minimal 路径从"约 1 秒可闻"退化为"完全无声" | 两个无文档依据的代码改动叠加：WAV 单声道混合广播 + `manual_connect` 取代 `helper_connect` |
| 2026-03-25 后期 | stock 路径（`tinyplay_ftm`）0.2s 瞬退 | `twrp-audio-play-test.sh` 脚本前置了对 AGM/FTM 有害的 mixer baseline 写入 |

### 三、已确认修复的问题

1. **soundtrigger VINTF 缺失**：recovery ramdisk 不含 `soundtrigger.qti.xml` → `audiohalservice.qti` 在 mandatory 注册点反复失败 → 补入后 HAL 稳定常驻。
2. **声卡 probe 时序**：冷启动第一次 `probe_sound` 太早，声卡未出现 → `twrp-audio-autoload.sh` 加重试逻辑后恢复。
3. **FE/BE PCM 格式错配**：默认 PCM config 改为 `48k/8ch/S32` 对齐 backend → 白噪音消除。
4. **缺 post-connect MFC**：把 pure C 成功链的 `helper_connect + tagged_info + configure_mfc` 移植到最小库 → runtime 步骤对齐。
5. **播放脚本回归**：`twrp-audio-play-test.sh` 对 AGM/FTM 路径跳过 `apply_speaker_mixer_baseline` → `tinyplay_ftm` 生命周期从 0.2s 恢复到 10s。

### 四、已排除的怀疑方向

- 写流过快/进程提前退出（逐秒日志证明 10s wall-clock 同步）
- Speaker baseline 回切（mixer 控件全程保持预期值）
- AW profile/monitor/switch 缺失（已确认 `Music + Enable + monitor on`）
- AW PA 在 1s 内主动 shutdown（内核日志证明 startup→shutdown 跨 10s）

- 显式 speaker kv 缺传（旧文档缩参数实验证明非硬门槛）

**当前状态**：
